# 智慧烟感预警系统 - 团队框架全景图与类说明规范

> **文档说明：** > 本文档是团队并行开发的“代码导航图”。为了杜绝代码冲突和逻辑混乱，本文档精准定义了当前仓库中**每一个包、每一个类的作用及归属**。
> **注意：** 未经沟通，请不要跨角色修改他人的核心类！

---

## 角色代号速查表
* **PM (技术型项目经理)**：负责核心数据模型、API契约与告警大脑。
* **BE1 (后端-物联侧)**：负责设备通信、假数据模拟、底层报文收发。
* **BE2 (后端-业务侧)**：负责接口 CRUD、鉴权安全、数据库交互、Websocket。
* **BE3 (后端-AI 侧)**：负责对接外部大模型与视觉复核 API。
* **FE1 & FE2 (前端组)**：负责管理后台界面与可视化大屏展现。

## 一、核心资源与配置文件 (Resources)

`src/main/resources/`
* `application.yml`  【PM/BE1/BE2 共同维护】 
  * **作用：** 系统的“总开关”。包含 MySQL、Redis、MQTT 连接信息。
  * **注意：** MQTT 的 Topic 契约 (`subscribe` 和 `publish-cmd`) 定义在此，**严禁在 Java 代码里硬编码 Topic**。
* `schema.sql`  【PM 掌控】
  * **作用：** 数据库建表脚本。定义了系统所有表结构（包含烟雾、温湿度字段）。项目启动时会自动执行。
* `data.sql`  【PM 掌控】
  * **作用：** 初始数据填充脚本。用于插入默认的管理员账号、系统配置字典等。

## 二、  后端 Java 源码 (Java Source)

**根目录：** `src/main/java/com/smartsmoke/`
 `SmartSmokeApplication.java`：Spring Boot 的主启动类。

### 1. `common` (公共组件层) 【全员只读，只允许调用】
* `Result.java`：统一的 API 响应封装类（包含 `code`, `msg`, `data`）。所有 Controller 必须返回该对象。
* `PageResult.java`：统一的分页响应封装类。
* `GlobalExceptionHandler.java`：全局异常处理器，兜底所有的报错，防止把系统堆栈直接抛给前端。

### 2. `config` (系统配置层) 【BE2 】
* `MqttConfig.java`：连接 EMQX 的底层配置（断线重连等）。
* `MyBatisPlusConfig.java`：配置 MyBatis-Plus 插件，如分页插件。
* `SaTokenConfig.java`：安全配置类，用于拦截所有 `/api/**` 的请求，校验 Token，放行登录接口。
* `WebSocketConfig.java`：开启 Spring WebSocket 支持的底层配置。

### 3. `controller` (HTTP 接口层) 【BE2 】
> **作用：** 接收前端发来的 HTTP 请求，校验参数，调用 Service，返回 Result 给前端。
* `UserController.java`：处理 `/api/auth/login` 等登录鉴权相关接口。
* `DeviceController.java`：处理前端设备列表查询、绑定设备的增删改查。
* `DataController.java`：处理历史环境数据（折线图）的分页与条件查询。
* `AlarmController.java`：处理历史报警日志的列表查询。

### 4. `mqtt` (物联网通信层) 【BE1 】
> **作用：** 纯粹的通道，负责和硬件设备（模拟器）打交道，**严禁在此处写业务判断**。
* `MqttConsumer.java`：**收发员**。负责监听设备上报的主题，将收到的 JSON 报文转换成 `SensorData` 对象，然后立刻丢给 `AlarmRuleEngine` 处理。
* `MqttPublisher.java`：**传令兵**。提供了一个 `sendCommand(deviceId, payload)` 方法。当 PM 决定要广播时，调用此类下发指令到真实硬件。

### 5. `rule` (告警规则大脑) 【 PM】
> **作用：** 整个系统最核心的心脏，处理从底层涌上来的所有传感数据流。
* `AlarmRuleEngine.java`：包含核心方法 `processData(SensorData)`。
  * **职责 1：** 调用 Service 把数据无条件存入数据库。
  * **职责 2：** 提取数据中的烟雾和温度值，进行**多模态判定**（如：烟雾>0.1且温度>60）。
  * **职责 3：** 判定起火后，呼叫 `AiService` 进行视觉复核，呼叫 `AlarmWebSocket` 弹窗。

### 6. `service` & `service/impl` (业务与 AI 服务层)
> **作用：** 处理具体的业务逻辑增删改查和第三方接口对接。
* **BE3 的领地：**
  * `AiService.java` & `AiServiceImpl.java`：系统的**防腐层**。封装了通过 Hutool 调用 `SmartJavaAI` 进行视觉复核的 HTTP 请求，并提供给 PM 调用。
* **BE2 的领地：**
  * `DeviceService.java` (Impl)：处理 `SmokeDevice` 相关的复杂业务。
  * `SensorDataService.java` (Impl)：处理传感数据的入库。
  * `UserService.java` (Impl)：处理用户校验等逻辑。
  * `AlarmRecordService.java` (Impl)：处理告警日志的生成和更新。

### 7. `entity` & `mapper` (数据持久层) 【 BE2 / PM 协同】
> **作用：** 与数据库中的表进行 1:1 映射并提供 SQL 访问能力。
* `SysUser.java` (Mapper)：系统用户表。
* `SmokeDevice.java` (Mapper)：设备信息表。
* `SensorData.java` (Mapper)：设备上报的温湿度、烟雾历史流水表。
* `AlarmRecord.java` (Mapper)：火警触发后的告警日志表。
* `AiReviewRecord.java` (Mapper)：记录 AI 视觉复核结果的日志表。
* `BroadcastRecord.java` (Mapper)：记录向下发广播指令的历史记录表。
* `ConversationLog.java` (Mapper)：前端调用 MaxKB 对话产生的日志。

### 8. `websocket` (实时消息推送层) 【BE2 领地】
* `AlarmWebSocket.java`：维护后端与前端大屏的长连接通道。当 PM 的规则引擎判定起火时，调用此类向前端瞬间推送红色告警指令。

## 三、 前端架构映射 (Frontend Structure)

前端 (FE1, FE2) 在 `src` 目录下严格按以下结构建立：

* `api/`：统一管理所有 Axios 请求。
  * `auth.js` (对应后端的 UserController)
  * `device.js` (对应后端的 DeviceController)
  * `data.js` (对应后端的 DataController)
* `views/`：页面级组件。
  * `dashboard/` (FE2)：数据大屏，引入 ECharts，并在 `onMounted` 里连接后端的 WebSocket。
  * `device/` (FE1)：设备管理，使用 Element Plus 的 Table 渲染 `device.js` 获取的数据。
  * `ai-chat/` (FE1)：仿微信聊天界面，直接对接 MaxKB 的开放对话接口。

## 四、核心工作流示例 (以“接收报文并触发警报”为例)

大家可以对照这个流程，看看自己的类在什么时候被执行：

1. **[模拟脚本]** 向 `smoke/SMK-01/data` 主题发送假数据 JSON。
2. **BE1** 的 `MqttConsumer` 收到数据 $\rightarrow$ 转成 `SensorData` 对象 $\rightarrow$ 调用 `AlarmRuleEngine`。
3. **PM** 的 `AlarmRuleEngine` 判断烟雾超标了！ $\rightarrow$ 调用 `SensorDataService` 存库 $\rightarrow$ 发现严重超标，需要复核！
4. **PM** 调用 **BE3** 的 `AiService.verifyFireVision()` 获取到结果。
5. **PM** 判断确认为火情，调用 **BE2** 的 `AlarmRecordService` 存入告警表。
6. **PM** 调用 **BE2** 写的 `AlarmWebSocket`，推送消息给前端大屏。
7. **FE2** 收到 WebSocket 消息，大屏立刻闪烁红色警报。
8. **PM** 调用**BE1** 写的 `MqttPublisher` 向 `smoke/SMK-01/cmd` 发出联动疏散指令。
