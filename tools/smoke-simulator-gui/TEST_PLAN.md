## 三、端到端全链路测试（模拟真实场景）

### 场景 3.1：管理员视角 — 设备上线→告警→处理→恢复

```
1. 启动后端 + EMQX + MySQL + Redis
2. 双击打开 SmokeSimulatorGUI.exe
3. 测试 MQTT 连接 → 成功
4. 从后端同步设备
5. 全选 3 台设备 → 启动心跳（间隔 5s）
6. 打开浏览器 http://localhost:8080/index.html
7. 确认大屏 3 台设备全部 🟢 在线 ← 验证：管理端实时状态
8. 选中 SD-001 → 选预设"🔴 火警"→ 点"📤 发送一次"
9. 大屏弹出告警通知！← 验证：MQTT→EMQX→MqttConsumer→AlarmRuleEngine→WebSocket→大屏
10. 浏览器进入告警列表 → 查看 AI 复核结果 ← 验证：告警自动触发 AI 复核
11. 确认/处理告警 ← 验证：告警状态流转（PENDING→CONFIRMING→CONFIRMED→CLOSED）
12. 停止 SD-001 心跳
13. 等待约 60 秒 → 大屏 SD-001 离线告警 ← 验证：Redis Keyspace 离线检测
14. 重启 SD-001 心跳 → 大屏恢复在线 + 离线告警自动关闭 ← 验证：恢复上线逻辑
```

### 场景 3.2：模拟器离线 → 后端恢复后数据同步

```
1. 启动心跳（所有设备在线）
2. 关闭模拟器（模拟模拟器崩溃）
3. 等待约 60 秒 → 大屏所有设备离线
4. 重新打开模拟器 → 启动心跳
5. 大屏设备全部恢复在线 ← 验证：MqttConsumer 自动注册/更新设备
```

### 场景 3.3：EMQX 宕机 → 模拟器自动重连

```
1. 所有设备心跳正常运行
2. docker stop smoke-emqx
3. 模拟器日志"MQTT 异常断开" + "断线重连已启动"
4. 等待 30 秒（观察重连间隔递增: 2s→4s→8s→16s→32s→60s）
5. docker start smoke-emqx
6. 模拟器自动重连成功，心跳恢复 ← 验证：零人工干预恢复
```

## 四、数据库验证 SQL

测试过程中可随时执行以下 SQL 验证数据是否正确入库：

```sql
-- 1. 查看设备列表及状态
SELECT id, device_id, device_name, status, battery, last_heartbeat 
FROM smoke_device ORDER BY id DESC LIMIT 10;

-- 2. 查看最近的传感器数据
SELECT sd.*, d.device_id 
FROM sensor_data sd JOIN smoke_device d ON sd.device_id = d.id 
ORDER BY sd.id DESC LIMIT 20;

-- 3. 查看最近的告警记录
SELECT id, device_id, alarm_type, alarm_level, alarm_status, alarm_time 
FROM alarm_record ORDER BY id DESC LIMIT 10;

-- 4. 查看 Redis 心跳 Key（需在虚拟机内执行）
-- docker exec smoke-redis redis-cli KEYS "device:heartbeat:*"

-- 5. 查看操作日志
SELECT * FROM operation_log ORDER BY id DESC LIMIT 10;
```

## 五、异常场景测试

| 场景 | 操作 | 预期 |
|---|---|---|
| 无网络启动 | 断网后双击 EXE | 窗口正常打开，设备列表可操作，MQTT 连接失败不崩溃 |
