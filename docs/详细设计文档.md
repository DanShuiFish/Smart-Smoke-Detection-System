## 1. 概述

### 1.1 项目背景

智慧烟感预警系统是一个面向家庭和公共场所的 AIoT（人工智能物联网）安全保障解决方案。在 15 天校内模拟期内，暂不依赖真实鸿蒙硬件，重点依托"模拟数据发生器 + 智能体构建 + 可视化大屏"三条软件主线完成能力储备。第 16 天起接入基地真实硬件时，要求云端和前端无需改动一行代码，体现软硬解耦的架构设计。

### 1.2 建设目标

构建一套端到端的全栈 AIoT 告警系统，核心数据链路为：

```
模拟传感器 → MQTT → 规则引擎判定 → AI 视觉复核 → WebSocket 推送大屏 → MQTT 广播疏散
```

具体目标：
- 感知层：Python 模拟器像素级复刻真实硬件报文协议，支持正常/告警/离线三种模式
- 传输层：MQTT（EMQX）承接待上报，Redis Keyspace 做离线检测
- 判定层：告警规则引擎秒级完成阈值判定，触发告警全链路
- AI 层：YOLOv8 视觉复核 + MaxKB RAG 智能问答，实现感知-决策-执行自主闭环
- 可视化层：管理端数据大屏 + 居民端仪表盘，WebSocket 实时推送

## 2. 总体架构设计

### 2.1 分层架构

系统采用 5 层架构，自下而上分别为：

1. **虚拟硬件层** — CLI 模拟器（smoke_simulator.py）+ GUI 模拟器（SmokeSimulatorGUI），像素级复刻硬件报文协议
2. **传输与缓冲层** — MQTT（EMQX）+ Redis TTL（心跳续期 + Keyspace 过期通知离线检测）
3. **核心处理层** — AlarmRuleEngine（规则引擎）+ 11 个 Service + 14 个 Controller + AlarmWebSocket + BroadcastService + MqttPublisher
4. **AI 智能层** — SmartJavaAI（YOLOv8 ONNX 视觉复核）+ MaxKB（RAG 知识库问答）
5. **多端可视化层** — 管理端大屏（ECharts 5.5）+ 居民端（ECharts 仪表盘）

### 2.2 物理部署拓扑

通过 MQTT 协议将设备端与云端解耦。Windows 开发机运行后端 Spring Boot 和 Python 模拟器。VMware Ubuntu 虚拟机通过 Docker 运行 EMQX、MySQL、Redis 三个中间件。后端通过 HTTP REST API 和 WebSocket 与前端通信，通过 MQTT 与模拟设备和硬件通信。

## 3. 系统模块详细设计

### 3.1 数据持久层

#### 3.1.1 实体设计（entity）

设计原则：
- 每张表使用 BIGINT AUTO_INCREMENT 主键
- 全部实体统一 is_deleted（0=正常，1=删除）逻辑删除
- 统一审计字段 create_by、create_time、update_by、update_time
- 时间字段使用 DATETIME，时区固定 +08:00

11 张业务表 + 1 张审计表：

| 实体 | 表名 | 核心字段 | 归属 |
|------|------|----------|------|

| SysUser | sys_user | username(唯一), password(BCrypt), role(RESIDENT/ADMIN), status(ENABLED/DISABLED) | PM |
| SensorData | sensor_data | device_id(FK), smoke_concentration(10,4), temperature(6,2), humidity(5,2), is_alert, collect_time | PM |

| ConversationLog | conversation_log | user_id, session_id, question(TEXT), answer(TEXT), user_rating(1-5) | PM |

| OperationLog | operation_log | user_id, operation_type, operation_target, request_ip, execution_time_ms | PM |

### 3.2 通信与配置层

#### 3.2.1 MQTT 通信模块（BE1）

职责边界：纯粹的通道层，只做报文收发和协议转换，不做业务判断。

MqttConfig.java — 连接 EMQX Broker（tcp://{host}:1883），自动重连，Broker 不可用时优雅降级（返回 null MqttClient 不阻断启动）。

MqttConsumer.java — 消息入口与分流：
- @PostConstruct 时订阅 smoke/+/data 和 smoke/+/heartbeat（Topic 从 application.yml 读取，禁止硬编码）
- handleMessage() 按 topic 分流：/heartbeat → handleHeartbeat()，否则 → handleDataReport()
- handleHeartbeat()：查 smoke_device 自动注册 → 更新状态 → Redis SET device:heartbeat:{code} EX 30
- handleDataReport()：设备身份转换 → 自动注册 → 时间戳安全降级 → 调用 AlarmRuleEngine.processData()

MqttPublisher.java — 指令下发：sendCommand(deviceId, payload) → smoke/{deviceId}/cmd，Qos=1

#### 3.2.2 实时推送模块（websocket — BE2）

AlarmWebSocket.java — 端点 /ws/alarm?token={token}
- onOpen()：URL 参数解析 token → 查用户角色 → 存入会话映射（解析失败时以 RESIDENT 保底）
- broadcast(msg)：全量广播
- broadcastByDevice(deviceId, msg)：按设备归属精准推送，ADMIN 全收，RESIDENT 只收绑定设备告警

#### 3.2.3 安全鉴权模块（config — BE2）

SaTokenConfig.java 拦截器配置：match /api/**，notMatch /auth/login、/auth/register、/health。is-global-filter: false 避免拦截静态资源。

#### 3.2.4 Redis 离线检测模块（config — BE2）

设计原理：利用 Redis Keyspace Notifications，当 device:heartbeat:{code} Key 过期时自动触发离线告警，避免扫表式检测。

RedisConfig.java — 注册 __keyevent@0__:expired 监听容器，启动时自动 CONFIG SET notify-keyspace-events Ex。

RedisKeyspaceListener.java — 过滤 device:heartbeat: 前缀的过期 Key，调用 deviceService.updateOffline() + alarmRecordService.createOfflineAlarm()。

离线检测全链路：设备 10s 心跳 → Redis SET EX 30 → 设备断网 → 30s Key 过期 → Redis 通知 → 监听器 → 更新 OFFLINE 状态 + 生成 DEVICE_OFFLINE 告警

#### 3.2.5 其他配置

MaxkbConfig.java — 读取 maxkb.base-url/api-key/application-id/timeout 配置属性。
WebSocketConfig.java — @Bean ServerEndpointExporter。
MyBatisPlusConfig.java — PaginationInnerInterceptor(DbType.MYSQL)。

### 3.3 核心业务层

#### 3.3.1 告警规则引擎（rule — PM）

AlarmRuleEngine.java 是整个系统的告警判定核心。

三阶段判定流水线：

阶段1 — 无条件存库：sensorDataService.save(data) 所有数据先落库

阶段2 — 多模态阈值判定：
- 烟雾超标判定：smokeConcentration > 0.1
- 温度超标判定：temperature > 60.0
- triggerAlarm() 内部：查 alert_threshold 表匹配阈值（设备级 > 全局级）→ 默认 MEDIUM → 生成 AlarmRecord（alarmCode 格式 ALG-{yyyyMMdd}-{HHmmssSSS}-{SM/TE}）→ AlarmWebSocket.broadcast() 推送
- HIGH/CRITICAL 级别进入阶段3

阶段3 — AI 视觉复核与广播联动：
- pickTestImage() 从 smart-smoke-models/test-images/ 随机选图
- aiService.verifyFireVision() → YOLOv8 推理
- 写 AiReviewRecord → 更新 alarm_status（CONFIRMED / CONFIRMING）
- 确认为火情 → BroadcastService.tryAutoBroadcast()

告警状态机：PENDING → CONFIRMED → RESOLVED / ARCHIVED / CLOSED，或 → CONFIRMING（待人工复核）→ CONFIRMED → ...

#### 3.3.2 设备管理模块（controller — BE2）

DeviceController.java — 7 个端点（CRUD + 统计 + 批量删除）

权限过滤方法 getVisibleDeviceIds()：ADMIN 角色返回 null（看全部），RESIDENT 返回绑定设备 ID 集合。创建设备时自动绑定创建者为 OWNER。

#### 3.3.3 告警日志模块（controller — BE2）

AlarmController.java — 6 个端点（列表/详情/确认/处置/归档/关闭），列表支持 status/type/level/deviceId/start/end 多条件过滤。

#### 3.3.4 告警阈值模块（controller — BE2）

AlertThresholdController.java — 5 个端点（CRUD）。device_id=NULL 表示全局默认阈值，查询优先级设备级 > 全局级。

#### 3.3.5 传感器数据模块（controller — BE2）

DataController.java — 2 个端点：latest（最新一条）+ history（分页 + 时间范围，pageSize ≤ 2000）。

#### 3.3.6 设备绑定模块（controller — BE2）

DeviceBindingController.java — 6 个端点，居民端 3 个（查我的设备）+ 管理端 3 个（全量分页/新增/解绑）。管理端列表返回 DeviceBindingVO 增强（含 deviceName + userRealName）。

#### 3.3.7 用户与鉴权模块（controller — BE2）

AuthController.java — 4 个端点（login/register/logout/me）。登录流程：查用户 → BCrypt 校验 → 状态检查 → StpUtil.login() → 更新登录信息 → 返回 token。

UserController.java — 7 个端点（CRUD + 改密码 + 重置密码）。

#### 3.3.8 大屏统计模块（controller — BE2）

DashboardController.java — 4 个端点（stats/realtime/alarm-trend/device-stats）。DashboardMapper 自定义 SQL 实现复杂聚合统计。

#### 3.3.9 广播联动模块（service — BE3）

BroadcastService.java / BroadcastServiceImpl.java — 双重触发模式：

createManualBroadcast() — 人工触发：参数校验 → resolveTargetAlarm() → 构建 MQTT 报文 → 下发 → 更新 sendStatus

tryAutoBroadcast() — 自动触发（资格校验链）：
1. alarmType 属于 {SMOKE_OVERFLOW, TEMP_OVERFLOW, FIRE}
2. isBroadcastSent = 0
3. alarmStatus 未关闭
4. alarmLevel 属于 {HIGH, CRITICAL}
5. 不在静默期（可配，默认 300s）
6. 辅助信号确认（CRITICAL 级别 / AI 已确认 / 烟雾温度双超标 / 连续 2 次异常）

BroadcastController.java — 3 个端点（创建/列表/详情）。

### 3.4 AI 智能层

#### 3.4.1 视觉复核模块（BE3）

AiServiceImpl.java — YOLOv8 ONNX 本地推理：

@PostConstruct initDetector() 加载 smart-smoke-models/best.onnx，CPU 推理，阈值可配置（默认 0.5）。模型加载失败时 fireSmokeDetector=null，视觉复核静默跳过。

verifyFireVision(imagePath)：调用 fireSmokeDetector.detect()，遍历检出对象，className 为 fire 或 smoke 且置信度达标则返回 true。异常或未检出返回 false（不阻断主流程）。

当前局限：测试期使用本地 test-images 目录图片，非真实摄像头流。

#### 3.4.2 智能问答模块（BE3）

MaxKB RAG 对话流程：
- getOrCreateChatId(sessionId)：查 Redis 缓存 → 命中返回 / 未命中 GET {baseUrl}/open 创建 → 写入 Redis（TTL 30min）
- chat(question, sessionId)：POST {baseUrl}/chat_message/{chatId}，body {message, stream:false, re_chat:true}，解析 data.content，空内容降级友好提示

ConversationController.java — 3 个端点（提问/历史/评分）。

### 3.5 模拟器层（BE1）

#### 3.5.1 CLI 模拟器（scripts/smoke_simulator.py）

5 种工作模式：正常 / 告警 / 离线 / 交互 / 新增设备。预置 5 台设备（SDS-001 ~ SDS-005），分布在不同楼栋。

正常数据范围：烟雾 0.01~0.03 mg/m3，温度 22~28 C。告警数据：烟雾 0.35 mg/m3，温度 68 C。

MQTT 报文：数据 Topic smoke/{code}/data。心跳 Topic smoke/{code}/heartbeat。与后端 DTO 字段完全对应（deviceId, smoke, temp, humi, bat, ts）。

#### 3.5.2 GUI 模拟器（tools/smoke-simulator-gui/）

Python Tkinter 桌面应用，5 大功能面板：设备列表 / 数据面板 / 连接控制 / 日志控制台 / 快捷操作。可独立打包为 exe。

SmokeSimulatorCore 多线程架构：工作线程按 interval 发送数据（随机/固定值模式），心跳线程独立发送，threading.Event() 控制线程优雅退出。配置持久化到 config.json 和 devices.json。

### 3.6 前端展现层

## 5. 接口设计

### 5.1 RESTful API

统一前缀 /api/v1/，响应格式 Result<T>：{ code, msg, data, timestamp }。分页响应包装 PageResult：{ page, pageSize, total, pages, records }。

| 设备 | /devices | 7 | CRUD + 统计 + 批量删除 |
| 绑定 | /bindings | 6 | 居民端3 + 管理端3 |
| 数据 | /data | 2 | latest + history |
| 告警 | /alarms | 6 | 列表/详情/确认/处置/归档/关闭 |
| 阈值 | /thresholds | 5 | CRUD |
| 大屏 | /dashboard | 4 | stats/realtime/alarm-trend/device-stats |

| 对话 | /conversations | 3 | 提问/历史/评分 |

| 配置 | /system-config | 2 | 列表/更新 |
| 健康 | /health | 1 | ping |

### 5.2 WebSocket 协议

端点：ws://{host}/ws/alarm?token={token}。推送策略：全量广播（broadcast）或按设备权限推送（broadcastByDevice）。管理员全收，居民只收绑定设备告警。

### 5.3 MQTT Topic 契约

| Topic | 方向 | Qos | 用途 |
|-------|------|-----|------|
| smoke/{code}/data | 设备→云端 | 0 | 传感器数据上报 |
| smoke/{code}/heartbeat | 设备→云端 | 0 | 心跳上报 |
| smoke/{code}/cmd | 云端→设备 | 1 | 广播指令下发 |

Topic 模板统一在 application.yml 中配置（mqtt.topics.subscribe 和 mqtt.topics.publish-cmd），禁止 Java 代码硬编码。

## 6. 核心业务流程设计

### 6.1 正常数据上报

模拟器 → MQTT smoke/+/data → MqttConsumer.parseDTO() → AlarmRuleEngine.processData() → save database → [不超标] 结束，大屏 30s 轮询拉取展示。

### 6.2 烟雾超标告警闭环（核心链路）

模拟器（超标数据）→ MQTT → MqttConsumer → AlarmRuleEngine 存 sensor_data → 阈值判定超标 → 生成 alarm_record（PENDING）→ AlarmWebSocket.broadcast() 大屏闪红 → 等级 HIGH/CRITICAL → AiService.verifyFireVision() YOLOv8 推理 → 确认火情 → alarm_status=CONFIRMED → BroadcastService.tryAutoBroadcast() → MqttPublisher 下发广播指令到 smoke/{code}/cmd

### 6.3 设备离线检测

设备心跳 → MqttConsumer.handleHeartbeat() → Redis SET device:heartbeat:{code} EX 30 → 设备断网 → 30s 后 Key 过期 → RedisKeyspaceListener.onMessage() → deviceService.updateOffline() + alarmRecordService.createOfflineAlarm() → WebSocket 推送离线告警

### 6.4 用户登录鉴权

POST /auth/login {username, password} → 查 sys_user → BCrypt 校验 → StpUtil.login() → 返回 token → 后续请求 Header Authorization=Bearer {token} → SaInterceptor 拦截校验

## 7. 安全设计

- **密码加密**：BCrypt 哈希存储，不存储明文
- **会话管理**：Sa-Token Token，前端 localStorage 存储
- **API 拦截**：match /api/**，放行登录/注册/健康检查
- **WebSocket 鉴权**：URL Token 解析 + 角色/绑定关系消息过滤
- **防越权**：居民端通过 getVisibleDeviceIds() 限制数据可见范围
- **逻辑删除**：统一 is_deleted 字段，不执行物理 DELETE

## 8. 部署与运维设计

### 8.1 Docker Compose

docker-compose.yml 编排 EMQX 5.3（端口 1883/8083/8084/18083）、MySQL 8.0（端口 3306，volume 持久化）、Redis 7.2（端口 6379）。

### 8.2 环境变量（配置外部化）

所有敏感配置通过环境变量注入，Spring @Value 配合默认值降级：DB_URL / DB_USERNAME / DB_PASSWORD（MySQL）、REDIS_HOST（Redis）、MQTT_BROKER_URL / MQTT_USERNAME / MQTT_PASSWORD（EMQX）、MAXKB_BASE_URL / MAXKB_API_KEY（MaxKB）、SMART_AI_MODEL_PATH（ONNX 模型路径）。

### 8.3 启动步骤

docker start smoke-emqx smoke-mysql smoke-redis → 修正 MySQL 时区（SET GLOBAL time_zone = '+08:00'）→ IDEA 运行 SmartSmokeApplication.java → 另开终端 python scripts/smoke_simulator.py --normal → 访问 http://localhost:8080

### 8.4 日志与监控

日志级别 com.smartsmoke: debug（开发期），SQL 日志 stdout。健康检查端点 GET /api/v1/health。操作审计写入 operation_log 表。
