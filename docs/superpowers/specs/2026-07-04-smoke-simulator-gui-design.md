# Smoke Simulator GUI 设计说明

日期：2026-07-04

## 1. 目标

为现有烟感模拟脚本提供一个独立的桌面图形界面工具，满足两类使用场景：

- 开发调试：快速切换模式、修改参数、查看日志
- 演示展示：无需命令行，界面直观、操作简单

该工具放在当前仓库中的新目录下，但与现有 Spring Boot 项目运行逻辑解耦，不依赖当前前端页面，不嵌入现有后端模块。

## 2. 目录结构

工具目录固定为：

```text
tools/smoke-simulator-gui/
  app.py
  simulator_core.py
  config.json
  devices.json
  requirements.txt
  README.md
```

说明：

- `app.py`：Tkinter 图形界面入口
- `simulator_core.py`：MQTT 通信与模拟逻辑
- `config.json`：连接配置与默认参数
- `devices.json`：默认设备与用户维护设备列表
- `requirements.txt`：依赖说明
- `README.md`：运行方式与常见问题

## 3. 总体方案

采用 Python `tkinter` 构建单窗口桌面工具，界面与模拟逻辑分离：

- 界面层负责用户输入、模式切换、日志显示、设备管理
- 核心逻辑层负责 MQTT 连接、数据生成、循环发送、离线模拟

现有 `scripts/smoke_simulator.py` 中的核心逻辑将作为参考进行拆分，而不是直接在原文件上堆叠 GUI 代码。

## 4. 功能范围

### 4.1 必做功能

1. 图形界面启动与关闭
2. MQTT 连接配置输入
3. MQTT 连通性测试
4. 默认设备加载
5. 用户新增、编辑、删除设备
6. 正常模式发送
7. 告警模式发送
8. 离线模式模拟
9. 日志实时输出
10. 配置文件持久化

### 4.2 本轮不做

1. 打包为 exe
2. 多窗口复杂导航
3. 自动发现 Broker
4. 与后端 API 联动读取设备列表
5. 图表化历史记录统计

## 5. 界面布局

主界面使用单窗口分区布局，建议从上到下、从左到右分为四块。

### 5.1 顶部：连接配置区

字段：

- Broker
- Port
- 用户名
- 密码

按钮：

- `测试连接`

状态显示：

- 当前连接状态
- 最近一次连接结果

### 5.2 左侧：设备管理区

内容：

- 设备下拉框
- 当前设备基础信息展示

按钮：

- `新增设备`
- `编辑设备`
- `删除设备`

设备字段：

- 设备编码 `device_code`
- 设备名称 `device_name`
- 楼栋
- 楼层
- 房间

### 5.3 中部：模式控制区

模式选项：

- 正常模式
- 告警模式
- 离线模式

操作按钮：

- `启动`
- `单次发送`
- `停止`

行为说明：

- 正常模式支持持续发送
- 告警模式支持单次发送，也可支持短时连续发送
- 离线模式通过停止心跳来模拟设备掉线

### 5.4 右侧：参数设置区

复选项：

- `使用随机值`

参数项：

- 烟雾值
- 温度
- 湿度
- 电量
- RSSI
- 数据发送间隔
- 心跳发送间隔

规则：

- 默认勾选 `使用随机值`
- 勾选时，正常模式使用随机值
- 取消勾选时，正常模式使用用户输入值覆盖随机值
- 告警模式默认填入一组超标值，但允许修改后发送

### 5.5 底部：日志输出区

内容：

- 连接信息
- 发送 topic
- 发送 payload
- 模式切换记录
- 错误信息

要求：

- 支持滚动
- 新日志自动滚到底部
- 错误信息与普通信息视觉区分

## 6. 数据与配置设计

### 6.1 config.json

用于保存：

- MQTT Broker
- Port
- 用户名
- 密码
- 默认模式参数
- 默认间隔配置
- 最近选择的设备
- 最近模式

### 6.2 devices.json

初始内置默认设备，同时允许用户修改后持久化。

默认策略：

- 第一次启动时写入默认设备列表
- 后续优先读取本地 `devices.json`

示例字段：

```json
[
  {
    "device_id": 1,
    "device_code": "SDS-001",
    "device_name": "1号楼1层101烟感",
    "building": "1号楼",
    "floor": "1F",
    "room": "101室"
  }
]
```

## 7. 运行逻辑

### 7.1 正常模式

- 若勾选 `使用随机值`：
  - 按现有脚本逻辑生成正常范围内随机值
- 若未勾选：
  - 使用界面参数输入值
- 按设定间隔持续发送数据
- 可同时发送心跳

### 7.2 告警模式

- 默认填充超标烟雾与高温值
- 用户可手工修改后发送
- 支持对当前选中设备发送一次告警报文

### 7.3 离线模式

- 对当前设备停止发送心跳
- 持续一定时间后完成模拟
- 在日志中提示离线倒计时和完成状态

### 7.4 停止逻辑

- 停止正常模式循环
- 停止心跳线程
- 保持界面可继续操作

## 8. 模块边界

### app.py

职责：

- 创建主窗口
- 创建各分区控件
- 绑定按钮事件
- 读取/保存配置
- 调用 `simulator_core.py`
- 刷新日志与状态

### simulator_core.py

职责：

- MQTT 客户端初始化
- 连接与断开
- 数据生成
- 单次发送
- 正常模式循环发送
- 心跳线程管理
- 离线模拟

要求：

- 不直接依赖具体界面控件
- 通过回调函数向界面输出日志

## 9. 错误处理

需要明确处理以下情况：

1. 未安装 `paho-mqtt`
2. Broker 地址不可达
3. 用户名密码错误
4. 端口非法
5. 未选择设备
6. 参数为空或格式错误
7. 发送过程中连接断开
8. 配置文件损坏或缺失

处理方式：

- 弹出提示框
- 同时写入日志区
- 尽量不让程序直接崩溃退出

## 10. 用户体验要求

1. 首次打开即可看到默认设备
2. 默认配置能直接用于常见演示
3. 日志信息清晰，不要求用户看命令行
4. 参数项命名尽量贴近业务
5. 常用操作按钮位置固定，减少误操作

## 11. 实施顺序

### 阶段一：最小可用版

1. 搭建 Tkinter 主界面
2. 接入 MQTT 连接与测试
3. 加载默认设备
4. 实现正常模式、告警模式、离线模式
5. 输出日志

### 阶段二：增强版

1. 设备新增/编辑/删除
2. 配置文件保存
3. 参数持久化
4. 状态栏与错误提示优化

## 12. 验收标准

满足以下条件视为完成：

1. 工具可独立运行，不依赖当前前端页面
2. 可在 GUI 中完成 MQTT 连接配置
3. 可选择默认设备并手动新增设备
4. 正常模式可随机发送，也可按用户设定值发送
5. 告警模式可一键发送超标数据
6. 离线模式可模拟停心跳
7. 日志区能清晰显示运行过程
8. 配置与设备列表能保存到文件

## 13. 推荐结论

本方案采用“独立目录 + Tkinter GUI + 配置文件”的实现方式，复杂度适中，既适合当前开发调试，也足够支撑演示展示，是本需求下最平衡的方案。
