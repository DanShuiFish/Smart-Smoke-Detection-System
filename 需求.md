# 智慧烟感预警系统
---

## 1. 项目介绍 (我们在做什么？)

[cite_start]**智慧烟感预警系统**旨在保障家庭和公共场所的消防安全 [cite: 2]。
项目的核心运转逻辑非常纯粹：
1. **平时（静默监测）：** 天花板上的烟感设备持续将“烟雾浓度”和“心跳（我还活着）”通过网络发给云服务器，系统在大屏上展示各区域安全状态。
2. [cite_start]**灾时（秒级响应）：** 一旦设备嗅到异常浓度（如超标），系统在 1 秒内捕获数据，前端大屏立刻闪烁报警；同时，系统自动唤醒 AI 大脑，调取事发地监控画面进行“视觉复核（看是否有明火）” [cite: 1, 2][cite_start]。确认火情后，通过网络向事发地下发“联动广播”指令，甚至通过 AI 智能体指导消防员进行处置 [cite: 1, 2]。

## 2. 项目分析 (需要什么技术底座？)

[cite_start]基于 15 天无真实硬件的限制以及企业级开发标准 [cite: 2]，我们的技术栈选型如下：

* **前端交互层 (UI/UX)：** `Vue 3` + `Vite` + `Element Plus` (快速搭建后台) + `ECharts` (数据大屏图表) + `WebSocket` (接收后端实时报警)。
* **后端服务层 (Server)：** `Spring Boot` (Java) 或 `Go` + `MyBatis-Plus` + `MySQL` (持久化存储) + `Redis` (热点数据与心跳过期检测)。
* [cite_start]**物联与中间件层 (IoT)：** `MQTT Broker` (EMQX/Mosquitto) 负责海量设备轻量级连接 + `Kafka/RabbitMQ` 负责瞬时高并发数据的削峰填谷 [cite: 2]。
* [cite_start]**数据发生层 (Mock)：** 使用 `Python (paho-mqtt)` 或 `Java` 编写脚本，作为“虚拟烟感器”发送模拟数据 [cite: 2]。
* [cite_start]**AI 与智能体层 (AI Agent)：** 接入 `MaxKB` 知识库构建问答智能体 [cite: 2] + [cite_start]对接 `SmartJavaAI` 第三方接口实现假定的视觉分析复核 [cite: 2]。

## 3. 项目拆解 (系统架构到底长什么样？)

为了便于理解，我们将系统垂直拆解为 5 个必须打通的层级：

1. **虚拟硬件层 (Mock Data Layer)：**
   * 本质是一个无限循环的脚本。
   * 任务：每隔 5 秒生成一段 JSON（包含设备号、浓度值），推送到 MQTT 的指定 Topic（主题）。提供一个“着火”按钮，按下发送超标浓度数据。
2. **传输接入层 (Transport Layer)：**
   * 本质是一个消息快递中转站（MQTT Broker）。
   * 任务：接收来自“虚拟硬件”的千万条消息。
3. **数据缓冲与处理层 (Processing Layer)：**
   * 本质是 Kafka 队列 + Spring Boot 后端。
   * 任务：把 MQTT 的消息转存到 Kafka 排队，后端消费队列。如果是正常数值，平缓存入 MySQL；如果是超标数值，立即触发告警引擎。
4. **AI 智能赋能层 (AI Layer)：**
   * 本质是对接大模型和视觉 API。
   * [cite_start]任务：当后端告警引擎启动时，自动请求第三方 AI 接口验证明火；同时为前端提供基于文档的消防知识问答 API [cite: 1, 2]。
5. **前端展示层 (Presentation Layer)：**
   * 本质是 Web 页面。
   * 任务：通过 HTTP 调用后端 API 查看历史；通过 WebSocket 保持与后端长连接，一旦后端说“着火了”，立刻弹窗变红。

## 4. 项目功能需求 (功能地图与技术对照)

[cite_start]结合产品原型要求，系统需实现以下核心功能模块 [cite: 1]：

| 模块 | 功能名称 | 业务场景描述 | 运作流程 | 核心技术栈 |
| :--- | :--- | :--- | :--- | :--- |
| **设备监控** | 烟雾浓度监测 | 实时展示虚拟设备发来的数值 | 脚本发送 -> MQTT接收 -> 后端入库 -> WebSocket推至前端大屏 | MQTT, WebSocket, Spring Boot |
| **数据展现** | 历史浓度趋势 | 以折线图展示过去一小时的浓度变化 | 前端发请求 -> 后端查 MySQL -> 前端 ECharts 渲染折线图 | ECharts, MyBatis-Plus |
| **告警管理** | 烟雾超标告警 | 浓度超过预设阈值（如 0.10）触发通知 | 后端判别超标 -> 存入日志表 -> WebSocket 推送 -> 前端响铃闪烁 | 规则引擎逻辑, WebSocket |
| **告警管理** | 设备离线告警 | 脚本停止发送心跳后，系统感知设备“死机” | 后端依靠 Redis 过期机制检测心跳断开 -> 触发告警通知用户 | Redis 键空间通知 / 定时任务 |
| **设备管理** | 设备总览与绑定 | 查看所有设备列表，管理员可录入新设备 | 后端提供标准的增删改查(CRUD) API -> 前端用 Element Plus 表格展示 | Element Plus, Spring Boot REST |
| **系统管理** | 告警日志追溯 | 以列表形式查看历史上所有的报警记录 | 后端提供带分页的 API -> 前端提供日期、类型检索面板 | 数据库分页查询 (PageHelper) |
| **AI 联动** | 视觉复核 | 告警触发后，联动模拟监控摄像头确认真伪 | 告警产生 -> 后端调 SmartJavaAI 接口 -> 返回"存在明火" -> 归档 | Http请求封装, AI 接口对接 |
| **AI 联动** | 警情智能问答 | 消防员通过对话框了解报警点处置预案 | 导入《消防手册》至 MaxKB -> 暴露 API -> 前端画聊天框对接 | MaxKB 平台, LLM 提示词工程 |
| **设备控制** | 联动广播下发 | 确认火情后，向设备反向发送广播指令 | 后端向 MQTT 的指令 Topic (如 `/cmd/broadcast`) 发送控制报文 | MQTT 发布(Publish) |

## 5. 项目成员分工 (每个人该做什么，怎么做？)

[cite_start]依据 6 人团队标准（1 PM + 2 前端 + 3 后端），实行严格的角色责任制 [cite: 2]：

### 1. 项目经理 (PM)
* **角色定位：** 大脑、粘合剂、进度监工。
* **具体任务：**
  * 在 Teambition/Gitee 建立 Kanban（看板），将功能模块拆解为具体的待办任务（Task）。
  * 制定接口规范文档骨架（推动前后端尽早定下 API 格式，使用 Apifox）。
  * [cite_start]组织每日 10 分钟站会，记录进度和卡点。统筹中期答辩 PPT 和汇报演示录制 [cite: 2]。

### 2. 前端开发 (FE1, FE2)
* **FE1 (后台系统与业务线)：**
  * **怎么做：** 负责开发后台管理框架，实现“设备列表”、“添加设备”、“告警日志列表”页面的画图与接口联调；独立完成“警情智能问答”的聊天对话框 UI 研发。
  * **产出：** 标准的管理系统页面，交互流畅。
* **FE2 (数据大屏与通信线)：**
  * **怎么做：** 聚焦一张图——“实时监控大屏”。负责集成 ECharts 绘制浓度动态折线图；负责编写 WebSocket 客户端代码，接收实时报警并做出强烈的视觉反馈（满屏红框、警报音）。
  * **产出：** 极具视觉冲击力的数据可视化大屏。

### 3. 后端开发 (BE1, BE2, BE3)
* **BE1 (物联底座基建)：** **全队前 3 天的关键先生**
  * **怎么做：** 使用 Docker 安装 EMQX (MQTT服务器)。编写 Python/Java 模拟器脚本，建立**“正常模式”**（定时发平稳数据）、**“异常模式”**（发高浓度数据）、**“断线模式”**（停发心跳）三个指令逻辑。
  * **产出：** 跑通从设备端到云端 MQTT 的全链路，保证数据不丢。
* **BE2 (核心业务研发)：**
  * **怎么做：** 设计 MySQL 数据库表结构（设备表、历史表、日志表）；引入 Kafka/Redis 做数据缓冲缓冲；编写供 FE1 调用的标准 CRUD 接口；编写核心报警判断逻辑和离线心跳检测逻辑。
  * **产出：** 健壮的后台服务体系，支撑所有页面数据流转。
* **BE3 (AI智能体研发)：**
  * **怎么做：** 重点攻坚 AI。部署或注册 MaxKB 平台，将提供的消防 PDF 转化为 RAG 知识库，暴露对话 API 供 FE1 接入；对接 SmartJavaAI，用 Java 写好图像复核的 HTTP 调用逻辑。
  * **产出：** 智能体工作流引擎，使项目具备真正的“智慧”。

## 6. 成员学习技术地图
必须**面向需求学习**。

### 前端组学习路线 (FE1, FE2)
* **基础框架：** 重点学 `Vue 3 (Composition API)` 和 `Vite`。B站搜“Vue3 尚硅谷/黑马速成”。
* **UI 与图表：**
  * FE1 专攻 `Element Plus` 组件库的 `Table` (表格) 和 `Form` (表单) 用法。
  * FE2 专攻 `ECharts` 官网的“折线图(动态数据更新)”和“仪表盘”实例配置。
* **通信技术 (FE2核心)：** 重点学 `WebSocket API` 或 `SSE`。学会在前端代码中 `new WebSocket(url)` 建立连接，并在 `onmessage` 中处理后端推送。

### 后端物联组 (BE1)
* **MQTT 协议深度：** 理解 `Broker`, `Topic`, `Payload`, `QoS 0/1/2` 的含义。
* **脚本编程：** 学习 Python 的 `paho-mqtt` 库。核心是掌握如何封装一段合法的 JSON，如 `{"deviceId":"SN001", "smoke":0.15, "ts":1699920000}`。

### 后端业务组 (BE2)
* **Web 框架：** `Spring Boot 3` + `MyBatis-Plus`。掌握三层架构 (Controller, Service, Mapper)。
* **心跳离线机制 (进阶)：** 学习 `Redis` 的生命周期 (TTL)。当收到心跳包，在 Redis 存入 `setex device:001 30 true` (30秒过期)。学习如何利用 `Redis Keyspace Notifications` (键空间通知) 在设备断联时自动触发告警代码。
* **并发削峰：** 了解 `Kafka` 或 `Redis Stream`，学会最基础的发布/消费模型。

### 后端 AI 组 (BE3)
* **RAG 原理：** 了解大模型“检索增强生成”的工作方式。学习使用 MaxKB 平台导入文档和调试 Prompt。
* **API 交互：** 学习 Spring Boot 中的 `RestTemplate` 或 `WebClient`，熟练掌握如何从 Java 后端向外部第三方 AI 接口发起 POST 请求并解析返回的 JSON。

## 7. 优化方案

1. **极致的“零代码修改”硬软过渡设计 (架构含金量)**
   * **痛点：** 很多学生团队前 15 天用一套假数据格式，第 16 天接到真硬件后发现解析报错，导致代码大面积重构。
   * **优化方案：** 在第一天，**PM和BE1必须拿到真实硬件（Hi3861等）的物模型说明书**。模拟脚本发出的 JSON 必须 1:1 像素级复刻真实硬件的报文格式。答辩时，一键停掉脚本，接入真实设备而云端无需修改一行代码，这是极高工程素养的体现。
2. **优雅的离线判定：告别低效的定时扫表 (后端含金量)**
   * **痛点：** 判定设备离线，通常做法是写个定时任务每秒去数据库查 `if(当前时间 - 最后心跳时间 > 30s)`。当设备量上万时，数据库会崩溃。
   * **优化方案：** 使用 **Redis 键空间过期通知**。设备每次发心跳，就在 Redis 重置一个 30 秒倒计时的 Key。一旦 30 秒没发，Redis 自动销毁该 Key 并触发事件，Spring Boot 监听到事件后生成报警日志。这种机制体现了对高并发场景的深刻理解。
3. **基于 Agent 的自主联动闭环 (AI 含金量)**
   * **痛点：** 把 AI 当作简单的“你问我答”聊天机器人。
   * [cite_start]**优化方案：** 让 AI 成为“决策大脑” [cite: 2][cite_start]。当超标告警发生时，不再是用 IF-ELSE 写死广播下发，而是将现场数据和第三方视觉复核结果打包，作为一个“事件”推给后台的“调度智能体” [cite: 2]。智能体基于预设的规则链思考后，自主决定是否向特定 Topic 发布广播控制指令。
