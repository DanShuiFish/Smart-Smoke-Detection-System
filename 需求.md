# 智慧烟感预警系统 - 15天模拟期全栈工程实施白皮书

> [cite_start]**文档说明：** > 本文档是基于「端-智-云一体化软件综合设计」的行动总纲领 [cite: 29][cite_start]。在校内的前 15 天（真实环境模拟期），我们暂不依赖真实鸿蒙硬件，重点依托“模拟数据发生器 + 智能体构建 + 可视化大屏”三条软件主线完成能力储备 [cite: 37]。
> **特别注意：** 本项目采用 Technical PM（技术型项目经理）驱动模式，PM 将深度下沉至核心数据建模与规则引擎开发，实现全局绝对掌控。

## 1. 项目介绍 (项目蓝图与核心闭环)

[cite_start]**智慧烟感预警系统**是一个面向家庭和公共场所的 AIoT（人工智能物联网）安全保障解决方案 [cite: 76, 79]。

在当前的 15 天模拟期内，系统运作的核心逻辑如下：
* [cite_start]**静默感知：** 系统通过 Mock 脚本模拟真实烟感传感器，源源不断地向云端发送环境数据（如烟雾浓度 0.02 $mg/m^3$）与心跳信号 [cite: 50]。
* [cite_start]**云端流转与展示：** 后端接收数据并削峰平谷，大屏实时渲染设备地图与历史决策轨迹 [cite: 52]。
* [cite_start]**秒级智能闭环（核心）：** 当模拟器发送“超标”数据时，系统需在秒级完成判定，触发告警 $\rightarrow$ 联动 SmartJavaAI 进行视觉复核（确认为明火） $\rightarrow$ 联动 MaxKB 智能体给出消防建议 $\rightarrow$ 下发广播指令进行人员疏散 [cite: 51, 60]。

## 2. 项目分析 (全栈技术底座选型)

[cite_start]根据 5R（真实环境、真实项目、真实工作、真实人才、真实成果）体系标准 [cite: 39]，我们为系统规划了企业级的主流技术栈：

* [cite_start]**物联网传输层：** `MQTT` / `TCP` / `WebSocket` [cite: 34, 50]。作为端云双向通信的轻量级通道。
* [cite_start]**高并发中间件：** `Redis Stream` [cite: 34, 50]。作为消息缓冲池，确保海量并发数据不冲垮数据库。
* [cite_start]**后端微服务架构：** `Spring Boot` (Java) + `MyBatis-Plus` + `MySQL` + `Redis` [cite: 64]。支撑核心业务接口与高频缓存。
* [cite_start]**AI 智能化平台：** `MaxKB` (知识库构建与大模型编排) [cite: 51, 61] + [cite_start]`SmartJavaAI` (视觉识别复核) [cite: 60]。
* [cite_start]**前端全栈展现：** `Vue 3` + `Element Plus` (后台业务开发) + `ECharts` / `DataEase` / `Three.js` (2D/3D可视化与大屏) [cite: 34, 52]。

---

## 3. 项目拆解 (五层系统架构模型)

为保障 6 人团队高效并行，我们将系统自下而上拆解为 5 个技术切面：

1. [cite_start]**虚拟硬件层 (Simulated Data Layer)：** 纯脚本构建的 Mock Server，按 MQTT 协议定时推送 JSON 格式的传感器数据 [cite: 50]。
2. [cite_start]**传输与缓冲层 (Transport & Buffer Layer)：** 由 MQTT Broker 接收报文，直接打入Redis Stream 进行排队 [cite: 50]。
3. **数据处理与规则层 (Core Processing Layer)：** 后端程序消费队列，记录历史数据。内置**规则引擎**实时监听，若浓度越界，立刻阻断常规流程并升级为“告警事件”。
4. [cite_start]**智能体与 AI 层 (AI Agent Layer)：** 承接告警事件，驱动感知-决策-执行闭环（调用视觉模型、RAG问答、生成广播指令） [cite: 51]。
5. [cite_start]**多端可视化层 (Visualization Layer)：** 分为 PC 管理后台与 PC 数据大屏，通过 API 查询历史，通过 WebSocket 实时捕获推送告警 [cite: 52]。

## 4. 项目功能需求 (业务映射与技术支撑)

结合《用户故事清单》，详细的需求落地转化如下表：

| 模块 | 业务功能 | 场景描述 | 核心落地技术栈 |
| :--- | :--- | :--- | :--- |
| **设备监测** | 烟雾浓度监测 | 实时展示虚拟设备发来的数值 | Python Mock脚本, MQTT, WebSocket |
| **数据展现** | 历史浓度趋势 | 折线图展示过去时间段的浓度变化 | ECharts, Spring Boot, MySQL分页 |
| **告警引擎** | 烟雾超标告警 | 浓度超过预设阈值触发全网通知 | Java 规则引擎判定, WebSocket 长连接推流 |
| **告警引擎** | 设备离线告警 | 设备心跳中断后触发故障通知 | Redis 过期时间(TTL) 与键空间通知 (Keyspace) |
| **后台管理** | 设备总览与绑定 | 查看全量设备状态，增删改查(CRUD)绑定 | Vue3, Element Plus, Spring Boot RESTful API |
| **后台管理** | 告警日志追溯 | 支持按时间、状态检索的历史报警列表 | MyBatis-Plus 分页插件 (PageHelper) |
| **AI 联动** | 视觉复核 | 告警时调用监控画面，AI识别明火形态 | SmartJavaAI (视觉技术), HTTP 客户端调用 |
| **智能体** | 警情智能问答 | 通过对话获取报警点详情与处置预案 | MaxKB 平台, RAG 检索增强生成, Prompt 提示词 |
| **联动控制** | 联动广播控制 | 确认火情后向 MQTT 下发疏散指令通知 | MQTT Publisher (下发 Topic) |

* **技术核心把控 (Tech Lead)：** * 操刀系统数据库底座（如 `device_info`, `alarm_log` 的表结构设计）。

### 2. 前端开发组 (FE - 2人)
* **FE1 (后台系统与业务线)：**
  * 职责：聚焦 Vue3 + Element Plus，负责设备管理、日志追溯页面的布局与 API 对接。
  * [cite_start]核心任务：实现“警情智能问答”交互界面（仿微信聊天框），完成与后端 AI 对话接口的联调 [cite: 74]。

* **BE1 (底层物联链路 - “感知与连接”)：**
  * [cite_start]职责：搭建并维护 MQTT Broker / Mock Server [cite: 50]。
  * 核心任务：编写 Python/Java 的“烟感模拟器脚本”。脚本需具备定时发送正常数据、一键触发高浓度报文、切断发送以模拟离线的多种模式。
* **BE2 (高并发与数据业务 - “躯干与持久化”)：**

* **BE3 (AI智能体集成 - “智慧大脑”)：**

## 7. 高级优化方案

为确保在校内中期答辩和基地终验中取得高分，我们必须体现深度的企业级工程思维，避免过分炫技：

1. **教科书级的“无缝虚实切换” (最强工程体现)：**
   * **现状痛点：** 多数团队的模拟期脚本 JSON 格式随意编造，第 16 天接入真实鸿蒙硬件时代码全面崩溃。
   * [cite_start]**优化方案：** PM 和 BE1 需提前获取 Hi3861 等基地真实硬件的数据上报协议（物模型） [cite: 55, 82]。我们的 Mock 脚本做到“像素级复刻”真实报文格式。答辩时，展示**云端和前端无需改动一行代码**，仅替换数据源即可打通硬件，体现顶级的“软硬解耦”架构设计。
2. **彻底抛弃“扫表式”检测，优雅的离线判定：**
   * **现状痛点：** 判别离线通常使用定时器每隔几秒扫描 MySQL 查时间差，当设备数达到上万时，直接造成数据库死锁。
   * **优化方案：** BE2 采用 Redis。每收到一次心跳，刷新一次带 30 秒有效期的 Key。一旦断网，Key 自动销毁，触发 Redis 过期事件立刻生成告警日志。体现对海量高并发业务场景的深度思考。
3. **前端 3D 数字孪生大屏升级：**
   * [cite_start]**优化方案：** FE2 若学有余力，不局限于 2D ECharts，引入 `Three.js` 构建极简的建筑 3D 模型 [cite: 52]。当告警发生时，3D 模型对应的楼层节点闪烁红色预警光，极大地提升项目的视觉震撼力。
4. **Agent 多智能体调度协同闭环：**
   * [cite_start]**优化方案：** 抛弃硬编码的 `if-else` 控制。构建“调度智能体 + 决策智能体”体系 [cite: 51]。当系统触发高压告警时，把判定权交给 AI，让 AI 根据现场反馈自动决定是否调用 SmartJavaAI 视觉接口、是否立刻下发广播，真正做到“感知-决策-执行”的自主 AIoT。
